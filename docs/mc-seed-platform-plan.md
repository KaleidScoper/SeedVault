## 1. MC 种子机制技术说明

### 1.1 种子的本质

Minecraft 的种子（Seed）是一个 64 位整数（Java 版）或 32 位整数（基岩版），作为伪随机数生成器（PRNG）的初始值，控制地形噪声、群系分布、结构放置等世界生成的全部过程。相同版本 + 相同端 + 相同种子 → 必然生成相同世界，这是确定性保证。

### 1.2 原版客户端 vs 服务端：完全一致

官方单人客户端和原版服务端使用相同的世界生成代码，同一个种子生成的世界逐块相同，无需区分。

### 1.3 Mod 环境（NeoForge / Forge）：分情况处理

| Mod 类型 | 对种子世界生成的影响 |
|---|---|
| 纯内容 mod（新物品、新生物、新机制，不改地形） | 不影响，地形与原版一致 |
| 添加新群系 / 新结构的 mod（Terralith、Oh The Biomes 等） | **完全破坏种子一致性**，世界面目全非 |
| 只改矿石 / 装饰生成的 mod | 地形骨架相同，细节（矿脉、植被）随机化 |

**已知问题**：NeoForge 本体在某些版本中存在生物群系生成差异 bug（如同一种子在原版生成丛林，NeoForge 却生成平原），这是框架层面的问题，与具体 mod 无关。

**平台设计结论**：

- 默认收录 **纯原版种子**，这是主体。
- Mod 整合包种子单独建立分类，**必须注明整合包名称 + 精确版本号**，不可与原版种子混用。
- 投稿表单中"Mod 环境"字段应设为必填且有明确提示。

### 1.4 Java 版 vs 基岩版：地形接近但结构不同

自 1.18 起 Mojang 做了大规模跨版本种子同步：

- **地形、群系布局**：相同种子在两端大体一致（前提：种子数值 < 4,294,967,296）
- **结构位置**（村庄、要塞、神庙等）：**仍然不同**，不可互用
- **山地、洞穴、矿石生成**：存在细节差异
- **种子空间**：Java 版 2⁶⁴，基岩版 2³²；Java 大数种子在基岩版无法使用

**结论**：Java 版和基岩版种子必须作为两个独立字段，不允许跨版本标注"通用"（除非经过实测验证且数值在安全范围内）。

### 1.5 各版本种子输入限制汇总

不同端、不同版本对种子值的合法范围和处理方式有不同的限制，这在设计投稿表单和种子数据存储时必须纳入考量。

#### 1.5.1 Java 版

| 限制项 | 说明 |
|---|---|
| **数字种子范围** | 64 位有符号长整数（Java `long`），范围 -9,223,372,036,854,775,808 ~ 9,223,372,036,854,775,807 |
| **文本/字符串种子** | 游戏调用 Java `String.hashCode()` 将字符串转为 **32 位有符号整数**，因此字符串种子只有约 42.9 亿种可能的世界 |
| **触发 hashCode 的条件** | 种子包含非数字字符（首位负号 `-` 或 `+` 除外），或数值超出 `long` 范围 |
| **seed = 0 的历史问题** | 在 1.18.2（22w03a）之前，输入 `0` 会被当作"留空"处理，实际生成随机世界。社区用 `creashaks organzine`、`pollinating sandboxes` 等 hashCode 碰撞为 0 的字符串绕过。**1.18.2 起已修复**，`0` 可正常使用 |
| **48 位限制（临时）** | 1.18 实验性快照期间种子一度限制为 48 位，21w41a 恢复为 64 位 |
| **首尾空格处理** | 1.18.2（22w03a）起，种子输入的首尾空格会被自动修剪（trim） |
| **留空** | 使用系统时钟生成随机种子 |

**对平台的影响**：
- 种子数值必须以**字符串**类型存储（`VARCHAR`），避免数据库整数溢出丢失前导零或符号
- 文本种子（如 `Glacier`）的哈希碰撞空间仅 42.9 亿，不同字符串可能映射到相同数值，这属于正常行为，不应视为重复投稿的判定依据
- 版本早于 1.18.2 的种子若值为 `0`，应在表单中给出提示说明

#### 1.5.2 基岩版

| 限制项 | 说明 |
|---|---|
| **32 位截断问题（旧版）** | 1.18.20.21 之前的基岩版，所有种子被强制转为 **32 位有符号整数**。输入大数 `013523053477933` 会被截断为 `-1798537171`（高位丢弃），Mojang 标记为 "Works as Intended" |
| **64 位支持** | 自 1.18.20.21（beta）起支持 64 位种子，范围与 Java 版一致 |
| **seed = 0** | 1.18.20.21 起，单个数字 `0` 可正常使用 |
| **留空** | 旧版使用 Unix 时间戳作为种子，更新后改为随机数生成 |
| **种子模板** | 基岩版提供 25 个预设种子模板，以及近 50 个生存/创造模式起始种子，这些是固定的内置值 |

**对平台的影响**：
- 基岩版投稿必须区分测试版本是否 ≥ 1.18.20（决定种子是 32 位还是 64 位语义）
- 32 位时代的基岩版种子，其实际存储值与用户输入值可能不同（截断），前端应在检测到大数值时主动警告
- 若用户输入的种子值超出 32 位范围但声称在旧版基岩版测试，应标记为"待核实"

#### 1.5.3 Legacy Console Edition（PS4 / Xbox One / Switch 等）

| 限制项 | 说明 |
|---|---|
| **种子位数** | 理论上 64 位，但不同平台实现有差异 |
| **输入方式** | 各平台虚拟键盘交互不同，部分版本有字符长度上限（如 Xbox 360 版限制约 20 字符） |
| **版本碎片化** | 传统主机版已停止更新（终止于 1.14 等效版本），与当前 Java/基岩版地形完全不兼容 |

#### 1.5.4 中国版（网易代理）

| 限制项 | 说明 |
|---|---|
| **敏感词过滤** | 文本种子可能经过网易的敏感词过滤系统，特定字符串种子可能被拒绝创建世界；过滤规则不公开，随时变更 |
| **平台差异** | 中国版 Java 端和基岩端均基于网易修改版代码，种子处理逻辑可能与原版存在未文档化的差异 |
| **版本滞后** | 中国版通常落后国际版数个版本，世界生成算法停留在较早版本 |

**对平台的影响**：
- 中国版种子应**单独标记** `edition = 'china_java'` 或 `'china_bedrock'`，不应混入国际版种子库
- 若平台未来考虑收录中国版种子，需额外说明版本滞后情况和兼容性注意事项

#### 1.5.5 种子输入限制对照速查表

| 端 / 版本 | 种子空间 | 数值范围 | 文本种子处理 | 特殊注意事项 |
|---|---|---|---|---|
| Java 版（所有版本） | 2⁶⁴ | -2⁶³ ~ 2⁶³-1 | hashCode → 32-bit | seed=0 在 <1.18.2 不可用 |
| Java 1.18 实验快照（临时） | 2⁴⁸ | 0 ~ 2⁴⁸-1 | hashCode → 32-bit | 已被 21w41a 撤销 |
| 基岩版 <1.18.20 | 2³² | -2³¹ ~ 2³¹-1 | 截断为 32-bit | 大数被静默截断，不可逆 |
| 基岩版 ≥1.18.20 | 2⁶⁴ | -2⁶³ ~ 2⁶³-1 | hashCode → 32-bit | 与 Java 版数值范围对齐 |
| Legacy Console | 2⁶⁴（理论） | 同 Java | 取决于实现 | 已停止更新，与现版本不兼容 |
| 中国版 Java | 2⁶⁴ | 同 Java | hashCode + 敏感词过滤 | 版本滞后、过滤规则不公开 |
| 中国版 基岩 | 取决于版本 | 取决于版本 | 截断/过滤 | 版本滞后、过滤规则不公开 |

#### 1.5.6 投稿表单中的种子输入校验建议

1. **前端实时校验**：检测种子值是否为纯数字、长度是否在安全范围内、是否超出对应端的数值上限
2. **基岩版 32 位警告**：若用户选择基岩版且测试版本 < 1.18.20，种子值超出 32 位范围时弹出明确警告："您输入的种子在旧版基岩版中会被截断为 [实际值]，两个种子生成的世界不同。"
3. **文本种子提示**：若用户输入的是非纯数字种子，提示"文本种子在 Java 版将被转为 32 位哈希值，只有 42.9 亿种可能，与数值种子的 2^64 空间不等价"
4. **seed=0 版本提示**：若种子值为 `0` 且测试版本 < 1.18.2（Java）或 < 1.18.20（基岩），提示"此版本中 seed=0 会被当作随机种子处理"
5. **存储策略**：数据库中 `seed_value` 字段使用 `VARCHAR(64)` 存储原始输入字符串，另设 `seed_numeric` 字段存储解析后的 64 位整数（可空），两者互补

### 1.6 模组加载器对种子世界生成的影响

模组加载器（Mod Loader）本身是否影响种子对应的世界生成，是一个容易被混淆的问题。核心结论是：**纯加载器（不安装任何世界生成类 mod）在绝大多数情况下不会改变世界生成结果，但存在已知例外**。

#### 1.6.1 FML（Forge Mod Loader / MinecraftForge）

| 限制项 | 说明 |
|---|---|
| **种子解析** | Forge 不修改原版的种子输入解析逻辑，从输入端看与原版一致 |
| **世界生成** | 纯 Forge（无任何修改世界生成的 mod）的地形、群系布局与原版一致 |
| **`seed=0` 的旧版 bug** | Forge 基于原版代码，运行在 1.18.2 以前版本的 Forge 服务端同样存在 `seed=0` 被当作随机种子的问题。**此 bug 来自原版，非 Forge 引入** |
| **已知问题：Feature 注册表强制同步（GH #7451）** | Forge 强制同步群系/地物注册表，当服务端有世界生成类 datapack 而客户端无对应 mod 时，会导致客户端无法连接。此问题不影响单机，但影响含自定义世界生成的模组服的兼容性 |
| **已知问题：维度种子回退异常** | 使用 datapack 添加自定义维度时，若未显式指定维度种子，Forge 的 `OverworldBiomeProvider` 不会回退到世界种子，导致自定义维度的种子实际为 `null`，生成结果不可复现 |
| **跨版本** | Forge 版本与原版 MC 版本强绑定，不同原版 MC 版本对应的 Forge 构建不可混用 |

**对平台的影响**：
- 纯 Forge（仅装客户端辅助 mod 如 JEI、小地图等，无地形 mod）的种子应与同版本原版一致，投稿时可将 `mod_env` 设为 "原版"
- 标注"基于 Forge 1.xx.x"的投稿，管理员应注意审核投稿者是否安装了世界生成类 mod

#### 1.6.2 NeoForge

NeoForge 是 2023 年从 Forge 分裂出来的分支，面向 1.20.1+ 版本。

| 限制项 | 说明 |
|---|---|
| **与原版的一致性** | 纯 NeoForge（无世界生成 mod）理论上与原版世界生成一致 |
| **已知问题：特定版本群系生成差异** | 部分用户报告在 NeoForge 某些版本中，同一种子在原版生成丛林，NeoForge 下却生成平原。此问题属于 **NeoForge 框架层面的 bug**，与具体 mod 无关，可能涉及群系/地物注册表初始化顺序差异 |
| **已知问题：`FlatLevelSource` 缓存（GH #2902）** | NeoForge 的超平坦世界生成器缓存了生成设置，导致 Biome Modifications API 对超平坦世界不生效 |
| **已知问题：世界生成 Tag 移除列表（GH #2319）** | NeoForge 在处理 `remove` 列表中的世界生成 tag 条目时有严格校验，移除不存在的条目会直接崩溃，与原版行为不一致 |
| **版本覆盖** | NeoForge 仅覆盖 1.20.1+，不向后兼容 Forge 1.19.x 及更早版本 |

**对平台的影响**：
- NeoForge 种子应作为独立的 `mod_env` 选项，与 Forge 区分，因为两者的框架差异可能导致种子不兼容
- 建议在详情页 NeoForge 种子的显眼位置显示"纯 NeoForge 环境实测"的提示
- 若某个种子被报告"在 NeoForge 下与原版不一致"，应标记为待核实并在种子描述中加注

#### 1.6.3 Fabric

Fabric 以轻量化著称，对原版代码的侵入性远小于 Forge/NeoForge。

| 限制项 | 说明 |
|---|---|
| **世界生成一致性** | 纯 Fabric Loader + Fabric API（不含任何修改世界生成的 mod）对世界生成**无任何影响**，种子与原版完全一致 |
| **Fabric API 的群系修改接口** | Fabric API 提供 `BiomeModifications` API（1.21+ 废弃，逐步迁移至 `DimensionEvents`），但此 API 仅在 mod 主动调用时才生效，默认不修改任何生成 |
| **ChunkRandom 种子截断** | Fabric API 内部使用的 `ChunkRandom` 类中，`setPopulationSeed`、`setCarverSeed`、`setRegionSeed` 等方法仅使用世界种子的**低 48 位**，这是原版 Minecraft 的行为，非 Fabric 引入 |
| **与 Forge/NeoForge 生态隔离** | Fabric mod 无法直接运行在 Forge/NeoForge 上（反之亦然）。同一世界生成类 mod 在不同加载器下的行为可能不同（取决于 mod 作者的适配质量） |
| **跨版本 mod 兼容性** | Fabric 本身版本更新较快，且 API 废弃周期短（如 BiomeModifications → DimensionEvents），旧 mod 在新 API 版本下可能出现世界生成偏差 |

**对平台的影响**：
- Fabric 纯原版（无地形 mod）种子可与 Java 原版种子合并管理，无需单独分类
- Fabric + 地形 mod 的种子应像 Forge/NeoForge 一样归类为"mod 环境"，注明 mod 列表及版本

#### 1.6.4 Quilt

Quilt 是 2022 年从 Fabric 分裂出来的分支，大部分 Fabric mod 可兼容运行。

| 限制项 | 说明 |
|---|---|
| **与 Fabric 的关系** | Quilt Loader 与 Fabric Loader 架构高度相似，纯加载器级别不修改世界生成 |
| **Fabric mod 兼容性** | 绝大多数 Fabric mod 可在 Quilt 上运行（反之不行），因此 Quilt + Fabric 地形 mod 的种子世界 <=> Fabric + 同样 mod 的世界生成结果一致 |
| **Quilt 专用 API** | Quilt 提供 QSL（Quilt Standard Libraries），mod 开发者若使用 QSL 特有的世界生成钩子，其生成结果可能与 Fabric 上同名 mod 存在细微差异 |
| **生态规模** | 原生 Quilt 生态较小，大多数用户通过 Fabric mod 兼容层运行 |

**对平台的影响**：
- Quilt 种子投稿极少，不建议在 MVP 阶段为其设立独立分类
- 若后续出现 Quilt 专用种子，可归入 Fabric 分类下并添加 "Quilt" 标签

#### 1.6.5 性能优化 Mod 对种子一致性的影响

一些广为使用的性能优化 mod 修改了底层世界生成或光照引擎，理论上不应改变种子结果，但存在边界情况：

| Mod 名称 | 功能 | 是否影响种子一致性 |
|---|---|---|
| **C2ME**（Concurrent Chunk Management Engine） | 并行化区块生成 | 官方声明不改变世界生成结果，但极少数版本存在并发竞态导致区块边界异常。若出现此问题，区块重新生成后应恢复一致 |
| **Lithium** | 优化游戏逻辑计算（不涉及世界生成） | **不影响**种子结果 |
| **Starlight** | 重写光照引擎（不涉及世界生成） | **不影响**种子结果（1.20+ 原版已内置类似优化） |
| **Sodium** | 渲染优化 | **与种子无关**，仅影响客户端渲染 |

**对平台的影响**：
- 添加了一条投稿提示："若安装了 C2ME 等区块优化 mod，请在截图时关闭，或至少额外附一张无优化 mod 下的截图作为参照"

#### 1.6.6 模组加载器种子兼容速查表

| 加载器 | 纯加载器（无地形 mod） | 已知框架级世界生成问题 | 种子空间 | 与原版一致性 |
|---|---|---|---|---|
| **Vanilla（Java 版）** | — | seed=0 bug（<1.18.2）、MC-239247（Feature 顺序不稳定） | 2^64 | ✅ 基准 |
| **Forge** | ✅ 与原版一致 | 种子回退异常（自定义维度）、Registry 强制同步 | 同原版 | ✅（纯加载器） |
| **NeoForge** | ⚠️ 部分版本群系异常 | GH #2902（FlatLevelSource）、GH #2319（Tag 校验） | 同原版 | ⚠️（部分版本有差异） |
| **Fabric** | ✅ 与原版一致 | API 废弃周期短，旧 API 可能漂移 | 同原版 | ✅（纯加载器） |
| **Quilt** | ✅ 与原版一致 | QSL 钩子与 Fabric API 细微差异 | 同原版 | ✅（纯加载器） |

> **核心原则**：加载器本身不决定种子的世界生成结果——**安装了什么 mod 才是决定性因素**。但 NeoForge 某些版本的已知 bug 是例外，应特殊对待。

## 2. 版本号差异分析

### 2.1 三层精度模型

版本兼容性分析需要区分三个层次，对种子有效性的影响截然不同：

#### 第一层：大版本跨越 → 种子往往完全失效

| 版本变化 | 核心变化 | 种子兼容性 |
|---|---|---|
| 1.17 → 1.18 | 世界高度从 256 扩展到 320，地形算法彻底重构，洞穴系统完全重做 | ❌ 完全不兼容 |
| 1.12 → 1.13 | 世界生成架构重写 | ❌ 完全不兼容 |
| 1.7 → 1.8 | 群系生成算法变更 | ❌ 完全不兼容 |

这类跨越后，同一种子生成的是完全不同的世界。

#### 第二层："游戏更新包"（Game Drop）内的小版本跨越 → 部分失效

Mojang 自 1.21 起将每个小版本作为独立更新包发布，每次都可能带来世界生成变化。**1.21 系列关键断层点**：

| 版本 | 更新包名称 | 关键世界生成变化 | 与前版兼容性 |
|---|---|---|---|
| 1.21.0 | Tricky Trials | 引入试炼室（Trial Chamber） | — |
| 1.21.1 | 补丁版本 | 无世界生成变化 | ✅ 与 1.21.0 兼容 |
| 1.21.2 | Bundles of Bravery | 苍白花园（Pale Garden）以实验性内容首次出现 | ⚠️ 断层开始 |
| **1.21.4** | **The Garden Awakens** | **苍白花园正式引入，成为独立地上群系** | ❌ 与 1.21.1 不兼容（苍白花园位置、大小全部不同） |
| **1.21.5** | Spring to Life | 进一步调整世界生成，苍白花园分布权重变化 | ⚠️ 与 1.21.4 存在可见差异 |

**实际意义举例**：一颗在 1.21.1 中出生点旁边有苍白花园的"卖点种子"，放到 1.21.4 里可能根本找不到该群系，或位置偏移数千格。这类种子不能标注"1.21 通用"。

#### 第三层：小补丁版本（x.y.z → x.y.z+1）→ 一般兼容

Mojang 在纯 bugfix 补丁版本（如 1.21.3 → 1.21.4 中的热修复）间通常维持世界生成一致性。但仍建议以实测为准，不做无据的"向后兼容"保证。

### 2.2 版本兼容性标注规范

平台应提供两个版本字段：

- **精确测试版本**（必填）：投稿者实际测试的版本，如 `1.21.4`
- **已验证兼容范围**（选填）：如 `1.21.4 ~ 1.21.5`，需经实测，不可猜测填写

参考竞品 wisehosting 的做法，对每颗种子列出完整的已验证版本列表，以"版本徽章"形式展示。

### 2.3 基岩版版本号规范

基岩版使用三段式版本号（如 `1.21.60`），与 Java 版的语义完全不同，不可混用。平台应将两者的版本选择器完全分离。

## 4. 推荐项目架构

### 4.1 选型原则

- 面向 GitHub 开源，优先选择轻量、易自部署的技术栈
- 避免 MySQL（改用 SQLite）、Nginx（改用内置服务器或 Caddy）
- 前后端分离，方便社区贡献
- 图片存储使用本地文件系统或兼容 S3 的对象存储（如 MinIO、Cloudflare R2）

### 4.2 技术选型

| 层次 | 推荐方案 | 理由 |
|---|---|---|
| 前端框架 | **Vue 3 + Vite + TypeScript** | 你已熟悉，生态成熟 |
| UI 组件库 | **Naive UI** 或 **Element Plus** | 中文友好，文档完善 |
| 后端框架 | **FastAPI（Python）** | 轻量、自带文档、异步支持好 |
| 数据库 | **SQLite**（开发/小规模）→ **PostgreSQL**（可选升级路径） | SQLite 零配置，单文件，适合开源项目 |
| ORM | **SQLAlchemy 2.x + Alembic**（数据库迁移） | 成熟，迁移到 PG 无痛 |
| 图片存储 | 本地 `/uploads` 目录（开发）→ S3 兼容接口（生产） | 通过环境变量切换，零代码改动 |
| 认证 | **JWT**（HTTP-only Cookie） | 无状态，轻量 |
| 部署 | **单 Docker Compose 文件**，前端 build 后由 FastAPI 托管静态文件，或用 Caddy 反代 | 一键部署，无 Nginx |

#### 投稿页 `/submit`

**表单分步设计（3 步，减轻填写压力）**：

**第 1 步：基本信息**
- 种子数值（输入框，自动检测是否为合法数值，支持纯字符种子）
- 版本端（Java / 基岩版，单选，切换后版本号选择器联动刷新）
- 精确测试版本（下拉，按端分别维护版本列表）
- 世界类型（单选）
- Mod 环境（单选，选"整合包"后展开整合包名/版本输入框）

**第 2 步：截图与坐标**
- 截图上传（拖拽 / 点击，最多 5 张，第一张自动设为封面，可拖拽排序）
- 出生点坐标（X / Z，必填）
- 关键坐标（动态列表，可添加多条，每条包含：名称 + X/Y/Z）

- 种子标题（≤ 50 字，带字数提示）
- 种子描述（富文本 / Markdown，≤ 500 字）
- 标签选择（玩法标签必选 1 个，特性标签可多选）
- 兼容版本范围（选填，说明"经实测此种子在以下版本表现一致"）
- 提交前预览

**表单验证要点**：
- 种子数值唯一性检查（同版本+同版本端不允许重复投稿）
- 截图文件大小限制（单张 ≤ 5 MB，支持 JPG/PNG/WebP）
- 坐标值格式校验（整数）

#### 审核后台 `/admin`

- 待审种子列表（默认视图）

- 版本管理：维护各端的版本号列表（新版本发布后在此添加，自动同步到投稿表单的下拉选项）

- **种子数值复制**：详情页核心交互，按钮需足够大，Mobile 端尤其重要

- **空状态设计**：搜索无结果时展示友好提示 + 引导投稿

## 6. 其他重要议题

### 6.1 内容审核机制

开放投稿意味着必须有审核流程，建议采用**先审后发**模式：

- 新用户投稿进入 `pending` 队列，等待人工审核
- 积累一定信誉值（如 5 个被通过的投稿）后升级为"信任用户"，享有**快速通道**（审核优先级更高，或直接发布）
- 审核标准应公开：截图是否与描述匹配、版本信息是否填写准确、是否存在重复投稿

### 6.2 版本维护的持续运营成本

这是容易被忽视的隐性成本。Mojang 每隔数月就发布一次版本，每次版本更新后：

1. 需要手动在后台添加新版本号（低成本，几分钟）
2. 需要评估新版本是否改变了世界生成（需要有人实测，中等成本）
3. 如果世界生成有重大变化，需要在站内通知社区（低成本）

建议在项目 README 中明确说明版本维护规则，或建立一个"版本更新响应流程"文档，吸引社区协作维护。

### 6.3 种子去重与防刷

同一颗种子可以被无限投稿，需要策略防止：

- **精确去重**：seed_value + edition + tested_version 三者相同视为重复，在投稿时前端实时检测并提示"该种子已有投稿，可查看现有条目"
- **相似版本去重提示**：如 1.21.4 的某种子已存在，投稿 1.21.5 相同种子时提示"此种子已有 1.21.4 版本条目，请确认两版本表现是否一致"
- **刷赞防护**：同一用户对同一种子只能点赞一次，服务端校验

### 6.4 SEO 与分享优化

种子详情页的 URL 应该对搜索引擎友好，且分享到社交平台时有完整的 Open Graph 预览：

```
URL 示例：/seed/12345-古代城市出生点-1.21.4-java
OG 图片：自动取种子封面截图
OG 描述：自动截取种子描述前 120 字
```

### 6.5 关于竞品差异化的再确认

竞品（seedb.co.uk、MCSeedShare）的核心缺失：

- **无中文界面**：国内玩家使用体验差
- **无玩法向分类**：多以地形/群系为维度，缺乏"速通友好"、"建筑胜地"等玩法导向标签
- **无 Mod 整合包种子**：整合包社群的需求完全未被满足
- **无版本兼容性徽章**：不直观展示跨版本可用情况

这些正是本平台的差异化机会，也应是 MVP 阶段优先确保做好的功能。

### 6.6 MVP 范围建议

第一个可发布版本应包含且仅包含以下功能，避免过度设计：

**必须有**：
- 种子列表页（含基础筛选：版本端、版本号、玩法标签）
- 种子详情页（含截图、坐标、一键复制种子值）
- 投稿表单（完整字段，人工审核）
- 简单的用户注册/登录（仅用于投稿和点赞）
- 管理员审核后台

**可以后期迭代**：
- 评论系统
- 用户主页 / 关注系统
- 信誉系统
- 整合包种子专区
- 种子地图可视化（接入 Chunkbase API 或开源的 cubiomes-wasm）
- RSS / API 开放

### 6.7 种子地图可视化（未来功能）

竞品中最吸引人的功能之一是地图预览（如 Chunkbase、MineNest 的 Seed Map），可以在浏览器中直接查看群系分布。实现方案：

- **开源方案**：[cubiomes](https://github.com/Cubitect/cubiomes) 是一个 C 语言种子分析库，有 WebAssembly 编译版本，可在浏览器端直接运行，无需服务端渲染
- **接入成本**：中等，需要一定的 WASM 集成工作，建议作为 v2 功能
