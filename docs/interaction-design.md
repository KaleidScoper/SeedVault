## 1. 核心设计命题

### 1.1 这个网站不是什么

在开始设计之前，先排除几种直觉上相似但不合适的类比：

| 类比 | 为什么不合适 |
|------|-------------|
| **搜索引擎**（Google/Bing） | 种子不是通过关键词检索的。用户不会输入"出生点有村庄的种子"——种子是随机数，没有语义索引。唯一的文本搜索范围是标题和描述，这是辅助功能而非核心交互 |
| **论坛/BBS** | 种子投稿不是讨论帖。种子是离散的、结构化的数据条目，有明确的生命周期（提交→审核→展示），不依赖持续讨论维系价值 |
| **Wiki** | 种子数据不可编辑（一经审核锁定），不存在协作撰写场景 |

### 1.2 这个网站是什么

**一个以种子截图为核心视觉载体的筛选式发现平台。** 最接近的类比是：

- **Unsplash / Pinterest**：视觉优先的卡片瀑布流，浏览即发现，筛选代替搜索
- **Steam 创意工坊**：结构化条目的分类浏览，标签驱动，排序灵活
- **食谱网站**：每条内容有明确的"原料"（种子值、版本）和"成品"（截图），用户按条件筛选而非自由搜索

### 1.3 三条设计原则

1. **截图即内容**。截图是用户判断种子好坏的唯一依据，它应该占据最大的视觉权重。文字元数据是辅助决策信息，不是内容主体。

2. **筛选优先于搜索**。架构上围绕 faceted filtering（分面筛选）设计，搜索框降级为辅助工具（匹配标题/描述关键词）。

3. **种子值是最重要的交互对象**。用户在详情页的核心动作是"复制种子值 → 粘贴到 Minecraft"。这个路径必须是最短、最无障碍的交互。

## 2. 品牌展示与视觉叙事

### 2.1 命名的隐喻

"Seed Vault" 直接指向现实中位于挪威斯瓦尔巴群岛的**全球种子库（Svalbard Global Seed Vault）**——一座嵌入北极永久冻土层的混凝土设施，保存着全球农作物种子的备份。这个隐喻对 Minecraft 种子共享平台来说几乎是完美的：

| 现实种子库 | 本平台 |
|-----------|--------|
| 保存农作物种子，应对全球危机 | 保存 Minecraft 种子，防止遗失 |
| 深埋冻土之下，沉默而安全 | 轻量、可靠、持久化存储 |
| 全球研究者按需调取样本 | 玩家按标签/版本筛选索要种子 |
| 极简混凝土 + 冰蓝色调 | 网站应反映这种冷静、可信赖的美学 |

### 2.2 视觉方向

从这个隐喻出发，网站的视觉语言应该是：

- **冷色调主色**：冰蓝/钢蓝（`#3b82f6` → `#1e40af` 范围）。不是玩具般的亮蓝，是北极天空和冻土的冷蓝。
- **中性底色**：灰白至浅灰（`#f8fafc` → `#e2e8f0`），像斯瓦尔巴的混凝土地板和积雪表面。
- **暖色点缀（极少）**：琥珀/铜色（`#f59e0b`），呼应种子库入口那件著名的发光艺术装置《Perpetual Repercussion》，仅在关键 CTA（如"复制种子值"按钮）使用。
- **字体气质**：等宽字体用于种子值（暗示数据/代码/确定性），其余使用无衬线字体保持清晰。

这不是说网站要做成"冷冰冰的混凝土建筑"——而是从隐喻中提取颜色和气质，具体执行时仍然是温暖可用的 UI。Minecraft 游戏截图的暖色调（草绿、阳光、水体）会自然地为页面注入色彩，网站本身不需要额外鲜艳。

### 2.3 首屏品牌叙事

首屏需要在"立即展示种子内容"和"传达品牌标识"之间找到平衡。目前的首页布局（5.2 节）优先展示了种子卡片。在此基础上，一个轻量的品牌区域放在卡片网格上方：

```
（桌面端）
┌──────────────────────────────────────────────────┐
│  [导航栏]                                         │
├──────────────────────────────────────────────────┤
│                                                   │
│  ┌──────────────────────────────────────┐         │
│  │                                      │         │
│  │   ◈  S E E D   V A U L T             │         │
│  │                                      │         │
│  │   Minecraft 种子共享平台               │         │
│  │   发现、分享、复制你喜欢的 Minecraft 世界  │         │
│  │                                      │         │
│  │   [🏕️ 生存] [⚔️ 速通] [🏗️ 建筑] [💀 极限] │         │
│  │                                      │         │
│  │   灵感来自斯瓦尔巴全球种子库              │         │
│  │   —— 保存、发现、再次开启               │         │
│  │                                      │         │
│  └──────────────────────────────────────┘         │
│                                                   │
│  热门种子  ←──  无缝衔接种子卡片区                   │

│  │ ...  │ │ ...  │ │ ...  │ │ ...  │             │
```

**设计要点**：
- 品牌区视觉上克制——没有大幅 Hero 图，没有巨幅动画。就是一个干净的文字区块。
- "◈ S E E D   V A U L T" 使用宽松字间距（letter-spacing），传达"档案/目录/收藏品"的感觉。
- 品牌区在滚动时自然被卡片内容覆盖，不固定不悬浮，不阻止用户看种子。
- 在移动端，品牌区进一步精简为一行标题 + 分类入口，不浪费竖屏空间。
- 品牌区底部用一条细灰线过渡到种子卡片区，视觉上暗示"品牌介绍完了，下面是内容"。

### 2.4 常驻品牌触点

品牌不应只存在于首屏。其他触达点：

| 位置 | 展示形式 |
|------|---------|
| 浏览器标签页 | `Seed Vault - Minecraft 种子共享平台`（favicon 使用冰蓝色 ◈ 图标） |
| 导航栏左上角 | `◈ Seed Vault`（文字 Logo，可点击回到首页） |
| 页脚 | `Seed Vault · 灵感来自斯瓦尔巴全球种子库 · GitHub 开源` |
| Open Graph（分享预览） | 标题 + 描述 + 种子截图 |
| 404 / 500 页面 | "这颗种子似乎迷失在冻土中了"（温和的品牌化错误文案） |
| 投稿成功页 | "种子已存入库中 ✓"（呼应"种子入库"的隐喻） |

### 2.5 Java 版 / 基岩版主题色

之前版本选择绿（Java）/ 橙（Bedrock）存在如下依据：

- **Java Edition → 绿色（`#22c55e`）**：Minecraft Java 版的启动器、官网、社区文化中长期使用草方块绿。苦力怕、草方块、经验值都是绿色的。这不是强制规范，而是社区 15 年来的默认共识。
- **Bedrock Edition → 琥珀/橙色（`#f59e0b`）**：基岩版启动器 UI 自 1.2 起使用暖灰 + 橙色系，且"bedrock"本身是地质概念，暖色岩层比冷色更直觉。同时与品牌暖色点缀一致。

两个颜色在色盲模拟（protanopia / deuteranopia）下可区分，但不应仅依赖颜色——每个版本徽章必须同时显示文字（"Java · 1.21.4"、"基岩 · 1.21.60"）。

## 3. 信息架构

### 3.1 内容层级

```
种子库（全局）
├── 版本端维度：Java 版 / 基岩版
│   ├── 版本号维度：1.21.5 / 1.21.4 / 1.21.1 / ...
│   │   ├── 玩法标签：生存 / 速通 / 建筑 / 极限 / 挑战
│   │   ├── 特性标签：村庄 / 要塞 / 古代城市 / 稀有群系 / 孤岛 / ...
│   │   └── Mod 环境：原版 / 整合包
│   └── ...
└── 特殊视图：热门 / 最新 / 已验证
```

### 3.2 URL 结构设计

所有筛选状态编码在 URL query string 中，保证可分享、可收藏：

```
/browse                             默认浏览（最新版本 + 全部标签）
/browse?edition=java                 Java 版
/browse?edition=bedrock              基岩版
/browse?tag=survival                 玩法标签筛选
/browse?tag=village,stronghold       多标签组合（AND 逻辑）
/browse?version=1.21.4              精确版本筛选
/browse?sort=popular                 排序切换
/browse?edition=java&tag=survival&version=1.21.4&sort=newest
                                    组合筛选

/seed/42-古代城市出生点-1.21.4-java   种子详情（SEO 友好 slug）
/submit                              投稿页
/user/kalei                          用户主页
/admin                               审核后台
/login                               登录页
```

**设计决策**：`/browse` 而非 `/search`。URL 命名本身就是交互模型声明——我们在浏览筛选，不是搜索。

### 4.2 导航栏设计

**已登录状态（已验证 Minecraft 正版）**：
```
[Logo SeedVault]  [浏览] [投稿]         [🔍] [🧑皮肤头像▼]
                                                   ├ 我的投稿
                                                   ├ 我的收藏
                                                   └ 退出
```
导航栏头像使用 mc-heads.net 提供的 Minecraft 2D 面部头像（32px）。下拉菜单中可选显示游戏内 ID 作为身份标识。

**已登录状态（未验证 Minecraft 正版）**：
```
[Logo SeedVault]  [浏览] [投稿]         [🔍] [👤 默认头像▼]
```
使用 Microsoft 账户头像或首字母头像（方形小圆角，非圆形）。

**未登录状态**：
```
[Logo SeedVault]  [浏览] [投稿]         [🔍] [🔑 登录]
```
单一登录按钮，文案"登录"。"注册"二字不出现——不存在独立注册流程，首次登录即自动注册。

搜索图标始终可见，但点击后展开的是一个轻量搜索栏（匹配标题/描述），而非占据首屏的搜索框。

## 5. 首屏与免登录体验

### 5.1 首页设计目标

用户打开网站后的前 3 秒应该：
1. **看到种子截图**（不是文字、不是表单、不是登录框）
2. **理解网站用途**（"这是看 Minecraft 种子的地方"）
3. **能立即交互**（点击/滚动/筛选，无需任何注册）

### 5.3 首屏关键设计决策

**不做的事**：
- ❌ 首屏不放大型 Hero 搜索框（这不是搜索引擎）
- ❌ 首屏不放登录/注册 CTA（强制登录会杀死转化）
- ❌ 首屏不放冗长的项目介绍（用户来干活的，不是来读文档的）

**要做的事**：
- ✅ 首屏上方即展示种子卡片，零滚动即可看到内容丰富度
- ✅ 分类快捷入口用大图标/emoji + 文字，点击即跳转带筛选的浏览页
- ✅ 导航栏固定（sticky），滚动时始终可见
- ✅ 所有浏览、查看操作无需登录；仅点赞/收藏/评论/投稿触发登录

## 6. 种子卡片：原子交互单元

### 6.1 卡片信息层级

种子卡片是用户做决策的唯一依据。信息按重要性排列：

```
┌──────────────────────┐
│                      │
│    截图（16:9）       │  ← 视觉主体，占卡片 60% 高度
│                      │
├──────────────────────┤
│  种子标题             │  ← 主文字，1 行，加粗
│  简短描述（2行截断）   │  ← 辅助信息，灰色
│                      │
│  [Java 1.21.4]       │  ← 版本徽章（绿/橙区分端）
│  🏕️生存 🏔️奇观       │  ← 标签，最多 3 个
│                       │
│  👤投稿者  ❤️ 42  👁 1.2k │  ← 元信息行
└──────────────────────┘
```

### 6.2 交互行为

| 行为 | 触发方式 | 响应 |
|------|---------|------|
| 进入详情页 | 点击卡片任意位置 | 导航至 `/seed/:id`（新页面，非弹窗） |
| 快速复制种子值 | 悬停卡片时出现"复制"按钮 | 点击复制，Toast 提示"种子值已复制" |
| 快速点赞 | 点击心形图标 | 若未登录 → 弹出登录弹窗；已登录 → 即时切换状态 |

**设计决策：卡片悬停显示复制按钮**。这个快速通道让用户在浏览页就能获取种子值，无需进入详情页。对老用户（明确知道自己要什么）是效率提升；对新用户（需要看截图判断）也不干扰。

### 6.3 视图密度切换

浏览页提供视图切换（图标按钮，非下拉）：

```
[📱 大图] [📋 列表]
```

- **大图模式（默认）**：适合随意浏览、视觉发现
- **列表模式**：适合有目的性的快速扫描，一行显示种子值、版本、标签、点赞数，缩略图缩小

用户偏好存入 localStorage，无需登录即可记住。

### 6.4 版本徽章设计

## 7. 浏览与发现：为什么不是搜索引擎

### 7.1 核心论证

搜索引擎的两个前提在此不成立：
1. **用户有明确的信息需求并能用关键词表述**（"我想找出生点附近有村庄的种子"）——种子是随机数，没有语义索引，不存在这样的查询
2. **系统有可索引的全文内容**——种子元数据只有标题和描述两个短文本字段，全文搜索的召回率极低

用户在种子平台上的真实行为是：**"我想看看最近有什么好种子"** 或 **"我想要 1.21.4 的生存向种子"**。前者是浏览行为，后者是筛选行为。两者都不需要搜索框。

### 7.2 筛选器设计

```

```


- 排序选择编码在 URL 参数中

### 7.4 搜索框的降级角色

搜索框仍然存在，但被降级为辅助工具：

- **位置**：导航栏右侧的 🔍 图标按钮，点击后展开一个内联输入框（非独立页面）
- **匹配范围**：仅匹配 `title` 和 `description` 字段（即投稿者自填内容）
- **交互模式**：输入即搜索（debounce 300ms），不需要按 Enter
- **结果展示**：在筛选器已经生效的结果集中做文本匹配，本质是"当前筛选结果内的二次过滤"

**为什么保留搜索框**：
- 老用户记得某个种子标题的关键词时，快速定位
- 管理员搜索特定用户投稿
- 不占用首屏空间，作为辅助工具存在

### 7.5 随机发现

对于"随便看看"的用户，提供轻度随机发现功能：

```
[🎲 随机一个]
```

点击后从当前筛选条件下的结果中随机选取一个种子并跳转详情页。这不是核心功能，但满足了一部分用户的探索欲望（"不知道该选哪个，帮我挑一个"）。如果筛选条件过严导致 0 结果，随机功能自动忽略筛选条件。

## 9. 投稿漏斗设计

### 9.1 漏斗入口

投稿入口放在导航栏，所有页面可见。未登录用户点击"投稿"时：

1. 不直接重定向到登录页
2. 弹出提示："投稿需要登录，登录后您填写的内容不会丢失"
3. 用户登录/注册后 → 直接进入投稿页（不回到首页）

### 9.2 分步表单（Wizard 模式）

采用三步向导（已在架构文档中定义），以下重点补充交互细节：

#### 步骤指示器

```
  ① 基本信息  ────  ② 截图与坐标  ────  ③ 标题与标签

  (当前位置高亮，已完成步骤显示 ✓)
```

- 已完成步骤可点击返回修改（数据保留）
- 浏览器刷新/关闭时，已填写数据存入 localStorage 草稿，下次进入时恢复

#### 第 1 步：基本信息

最关键的精进点在于**种子值的实时校验反馈**：

```
种子值: [________________________]  ← 输入框

┌─────────────────────────────────────────────┐
│ ✅ 格式正确：64 位有符号整数，范围合法         │  ← 成功状态
└─────────────────────────────────────────────┘

┌─────────────────────────────────────────────┐
│ ⚠️ 此数值超出 32 位范围                        │
│    若在基岩版 < 1.18.20 中使用将被截断为 -xxx  │  ← 警告状态
└─────────────────────────────────────────────┘

┌─────────────────────────────────────────────┐
│ ❌ 该种子已存在                               │
│    已有相同版本端 + 相同版本号的投稿 [查看]     │  ← 重复状态
└─────────────────────────────────────────────┘
```

- 校验在输入 debounce 500ms 后触发
- 版本端切换后种子值不重置（用户可能先输入种子值再调整版本端）

#### 第 2 步：截图与坐标

- 拖拽区域显示虚线边框 + 上传图标 + "拖拽图片到此处或点击上传"
- 上传后预览缩略图网格，支持拖拽排序
- 第一张自动标记"封面"角标，可手动切换
- 坐标输入使用内联表单（X 输入框 + Z 输入框，Y 选填），点击"添加坐标"新增行

#### 第 3 步：标题与标签

- 标题输入框下方有字数提示（实时更新）
- 描述使用 textarea，支持 Markdown
- 标签选择使用 Chip（标签块）形式，可点击多选，选中的高亮

### 9.3 投稿草稿保存

```
localStorage key: "seedvault_draft"
expires: 7 天
内容: JSON { step, formData, lastUpdated }
```

- 每次表单值变更自动保存（debounce 2s）

## 13. 认证墙与渐进式登录

### 13.1 触发登录的操作

| 操作 | 匿名用户行为 | 登录后行为 |
|------|-------------|-----------|
| 浏览首页 | ✅ 直接访问 | 同 |
| 浏览种子列表 | ✅ 直接访问 | 同 |
| 查看种子详情 | ✅ 直接访问 | 同 |
| 复制种子值 | ✅ 直接复制 | 同 |
| 筛选/排序 | ✅ 直接操作 | 同 |
| 点击"点赞" | 🚫 弹出登录弹窗 | ✅ 即时点赞 |
| 点击"收藏" | 🚫 弹出登录弹窗 | ✅ 即时收藏 |
| 点击"评论" | 🚫 弹出登录弹窗 | ✅ 进入评论输入 |
| 点击"投稿" | 🚫 弹出登录弹窗 | ✅ 进入投稿页 |
| 点击"举报" | 🚫 弹出登录弹窗 | ✅ 进入举报流程 |

### 13.2 登录流程：单一 Microsoft OAuth 入口

不存在独立的"注册"和"登录"两条路径。只有一个按钮：**"使用 Microsoft 账号登录"**。点击后：

1. 浏览器重定向到 Microsoft OAuth 授权页（`login.microsoftonline.com/consumers/oauth2/v2.0/authorize`）
2. 用户选择 Microsoft 账户并授权
3. Microsoft 重定向回本站 callback
4. 后端自动执行 Xbox Live 认证 → XSTS 认证 → Minecraft 服务认证
5. 后端查询 `/entitlements/mcstore` 检查是否拥有 Java 版正版
6. 后端获取 `/minecraft/profile` 拿到 Minecraft UUID 和游戏内 ID
7. 首次登录 → 自动创建用户记录（无"注册表单"步骤）
8. 签发本站 JWT，设置 HTTP-only Cookie
9. 前端收到 callback → 显示登录成功状态

**用户体验**：用户只做一次选择（选择 Microsoft 账户并授权），其余全部自动化。

### 13.3 登录弹窗 vs 页面跳转

**弹窗模式**（用于点赞、收藏、评论）：
- 用户点击操作 → 弹出居中模态框 → 模态框内显示"使用 Microsoft 账号登录"按钮
- 点击按钮 → 整页重定向至 Microsoft OAuth（OAuth 不支持 iframe 内完成）
- 回调后弹窗自动关闭，原操作自动执行
- 优点：OAuth 回调后能回到操作现场

**页面跳转**（用于投稿）：
- 用户点击投稿 → 弹出模态框 → 点击"使用 Microsoft 账号登录" → OAuth 重定向
- 回调携带 `?redirect=/submit` 参数，登录后直接进入投稿页
- 投稿是独立的创作流程，跳转更自然

### 13.4 匿名用户的本地收藏

为未登录用户提供基于 localStorage 的临时收藏功能：

```
localStorage key: "seedvault_bookmarks"
内容: [42, 17, 108]  (种子 ID 数组)
```

- 未登录时心形图标正常显示，点击可收藏/取消
- 登录后，本地收藏自动合并到服务端（首次登录时触发）
- 提示："登录后将保留您的收藏记录"

## 14. 社交功能与社区信任

### 14.1 评论系统

- 评论排序：最早 → 最新（线性对话），不做嵌套
- 支持 Markdown（链接、代码块、粗体），不支持图片
- 评论者头像（Minecraft 皮肤头像或 Microsoft 头像，28px 方形小圆角）+ 游戏内 ID 或 display_name + 时间 + 内容
- 自己的评论可删除（软删除，保留"该评论已被作者删除"占位）
- 评论区在详情页下方，默认展开前 3 条，超出部分"加载更多"

### 14.2 信任信号

种子列表和详情页中需要传达信任信号：

```
✅ 已验证             ← 管理员/版主确认过的种子
👑 资深投稿者         ← 投稿通过数 ≥ 5 的用户
🆕 新投稿             ← 24h 内提交的种子
🎮 Java 版正版玩家    ← 系统验证 owns_java_edition，不可手动声索
```

- "已验证"徽章需要管理员手动标记
- "资深投稿者"系统自动计算（投稿通过数），无需手动标记
- "正版玩家"系统自动验证（来自 `/entitlements/mcstore`），用户无法伪造
- "正版玩家"的语义：投稿者确实拥有 Minecraft Java 版，其提交的种子版本信息因实际测试而更可信。未拥有者亦可投稿，仅无此标识

### 14.3 用户身份展示

登录用户的身份在各场景中展示：

| 场景 | 展示 |
|------|------|
| 导航栏 | 32px Minecraft 2D 面部头像（来自 mc-heads.net），未验证则用 Microsoft 头像 |
| 种子卡片的投稿者 | 24px 头像 + 游戏内 ID（如已绑定），否则用 Microsoft display_name |
| 用户主页 | 80px 头像 + 游戏内 ID + "正版玩家"徽章（如适用） |
| 评论区 | 28px 头像 + 游戏内 ID 或 display_name |

头像从不使用圆形裁切（`border-radius: 50%`）。Minecraft 皮肤头像为像素艺术（8×8 面部），圆形会破坏可读性。使用 `border-radius: 4px` 的方形小圆角。

### 14.4 举报流程

- 种子详情页有"举报"链接（低调设计：灰色小字，位于信息卡片底部）
- 点击 → 弹出举报表单（选择原因：截图不实 / 版本错误 / 重复投稿 / 不当内容 / 其他）
- 可填写补充说明
- 提交后反馈："举报已提交，感谢您的贡献"
- 不要求登录（但登录后举报会关联用户，便于管理员追问）
- 同一用户对同一种子多次举报合并为一条

## 15. 其他重要议题

### 15.1 性能感知设计

种子浏览页是图片密集型页面。交互上需要处理加载延迟：

- **渐进式图片加载**：先显示低分辨率缩略图（blurhash / LQIP），再替换为全分辨率
- **虚拟滚动**：当种子数量 > 100 时，使用虚拟列表（仅渲染可见区域的卡片）
- **图片懒加载**：使用 `loading="lazy"` + Intersection Observer polyfill
- **骨架屏**：首次加载时展示 8-12 个骨架卡片，让用户感知到内容即将出现

### 15.2 键盘导航

为桌面端重度用户提供键盘快捷操作：

| 快捷键 | 功能 |
|--------|------|
| `←` `→` | 详情页切换截图 |
| `Esc` | 关闭灯箱 / 关闭弹窗 |
| `Ctrl/Cmd + Enter` | 提交表单（投稿/评论） |
| `Tab` | 在筛选器中移动焦点 |

浏览页不做键盘快捷键（干扰原生滚动和 Tab 导航）。

### 15.3 SEO 与社会化分享

- 种子详情页使用 SSR/SSG 预渲染，确保搜索引擎可索引
- Open Graph 标签：
  - `og:image` = 种子封面截图
  - `og:title` = 种子标题
  - `og:description` = 种子描述前 120 字
- Twitter Card 同样支持
- 分享到 QQ/微信时，种子值作为纯文本预设（方便直接粘贴）

### 15.4 国际化准备

尽管 MVP 面向中文社群，数据结构应预留 i18n 空间：

- 标签名称使用 `tag_key`（如 `tag.survival`）+ 前端翻译
- UI 文本集中在 i18n 文件中（`.json` 或 `.yaml`）
- 用户生成内容（标题、描述、坐标名）不做机器翻译，保持原语言

### 15.5 黑暗模式

- 首发支持自动跟随系统主题（`prefers-color-scheme`）
- 提供手动切换（导航栏太阳/月亮图标）
- 截图在暗色背景上呈现效果更好（更接近游戏内观感）

### 15.6 种子数据的"时效性"

一个容易被忽视的 UX 问题：**旧版本种子是否仍有价值？**

- 浏览页默认展示最新两个大版本（如 1.21.x 和 1.20.x），而非全部版本
- 版本筛选器中旧版本默认折叠，点击"展开更多版本"查看全部
- 首页"热门"算法对旧版本种子做降权（超过 2 个大版本的种子权重 ×0.3）
- 这样确保首屏展示的是对当前玩家最有用的种子

### 15.7 管理后台的交互理念

管理后台不与普通用户争抢设计资源。核心原则：

- 功能完整但视觉极简（类似 WordPress 后台风格）
- 审核操作的最短路径：列表 → 预览（侧滑抽屉，不跳页）→ 通过/拒绝
- 批量操作：多选种子后批量通过/批量标记版本范围
- 操作日志：每个种子记录审核历史（谁、何时、做了什么）
